Message Metadata in ThoughtMachine

Overview

Messages stored in user_history (the append-only conversation log) can contain metadata fields beyond the standard role, content, tool_calls, and tool_call_id. These fields are used internally by the ThoughtMachine system for sequencing, rendering, pruning logic, and other features. They are generally not passed to the LLM (or are harmless if passed), but they are preserved in the session JSON for auditing and debugging.

Standard metadata fields

The following fields may appear on any message:

- seq (integer): A sequential index assigned to each message when it is appended. Used for ordering and debugging. Generated by Session._get_next_seq().

- created_at (string, ISO format): Timestamp when the message was added to user_history. Used for display and session persistence.

- is_system_notification (boolean): Indicates that the message is a system-generated notification (token warning, countdown expiry, context cleared). Used only in turn counting and summary insertion logic (see System Notifications document). Do not set this flag on normal user or assistant messages.

Fields specific to tool messages

On assistant messages that contain tool_calls:

- tool_calls (list): Standard OpenAI-compatible tool call objects, each containing id, type, and function.

On tool result messages:

- tool_call_id (string): Matches the id of the corresponding tool call.

- content (string): The output of the tool.

Fields used during summarisation (pruning)

On summary system messages (inserted by _apply_summary_pruning):

- summary (boolean): Always True. Indicates this system message is a summary of pruned conversation turns.

- pruning_keep_recent_turns (integer): The value of keep_recent_turns passed to SummarizeTool.

- pruning_discarded_msg_count (integer): Number of messages that were pruned (excluded from LLM context) before this summary.

- pruning_insertion_idx (integer): The index in user_history where the summary was inserted.

These fields are used for debugging and are not read by the LLM.

Guidelines for adding new metadata fields

1. Choose a descriptive, lower_snake_case name. Prefix with an underscore (e.g., _internal_state) if the field is strictly internal and should never be exposed.

2. If the field is intended only for internal logic (e.g., controlling turn grouping or context building), ensure that any code that copies messages to the LLM either strips the field or the LLM ignores unknown fields.

3. For backward compatibility, new fields should be optional. Old session files without the field should continue to work.

4. Document the field in this file and in any relevant component documentation.

5. If the field affects LLM behaviour (e.g., instructing the model to treat a message differently), consider whether a separate role or content pattern would be more appropriate.

Where metadata fields are stripped (or not)

- SummaryBuilder.build (in session/context_builder.py) currently copies all fields from user_history to the LLM context. It does not strip any metadata. This is acceptable because most LLM APIs ignore unknown fields. However, if a future LLM provider rejects extra fields, a filtering step should be added.

- The Qt GUI reads user_history directly and uses only role and content (plus is_system_notification for styling). Other fields are ignored.

- The session persistence (to_persistable_dict and from_persistable_dict) preserves all fields present in the message dictionary.

Backward compatibility

Adding a new metadata field to newly created messages does not break old sessions. When loading an old session, the field will be absent. Code that reads the field should use msg.get('field_name') with a sensible default (e.g., False for boolean, None for optional values).

Examples

A typical user message after being stored:

{
  "role": "user",
  "content": "What is the weather?",
  "seq": 42,
  "created_at": "2026-04-22T10:30:00"
}

A system notification with metadata:

{
  "role": "user",
  "content": "[SYSTEM NOTIFICATION] Token countdown expired...",
  "seq": 98,
  "created_at": "2026-04-22T10:35:00",
  "is_system_notification": true
}

A summary system message inserted during pruning:

{
  "role": "system",
  "content": "Summary of previous conversation: ...",
  "seq": 99,
  "created_at": "2026-04-22T10:36:00",
  "summary": true,
  "pruning_keep_recent_turns": 3,
  "pruning_discarded_msg_count": 33,
  "pruning_insertion_idx": 34
}
