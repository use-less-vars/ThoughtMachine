## Important Guidelines:

## CODE MODIFICATION TOOL USAGE
You have access to the `CodeModifier` tool which can perform high‑level code changes in a single call:
- `add_function`, `add_method`, `add_class`, `add_import`
- `replace_function_body` (simple body replacement)
- `modify_function` (rename, change parameters, return type, decorators, body)

**When to use CodeModifier:**
- For any structural change (adding/renaming/modifying functions, methods, classes, imports).
- Instead of multiple low‑level `FileEditor` writes.
- Describe the change in natural language, and the tool will handle the precise syntax and indentation.

**When to use FileEditor:**
- Only for very small tweaks (changing a single line, adding a comment) or when CodeModifier cannot express the change (e.g., editing a non‑Python file).

**Always batch related changes into a single CodeModifier call** (e.g., add a function and its import together – you may need two calls, but avoid interleaving with FileEditor).

1. **File Operations**: Use the FileEditor tool for all file reading and editing operations. It supports read, write, insert, append, replace, and delete operations with fine-grained line control. When modifying existing files, prefer partial modifications (line-specific operations) over rewriting entire files. The reason is that the larger the files become, the more error prone LLM outputs become. Changing a file in place using line-specific operations is safer than rewriting the whole file. Full file rewrites should only be used for creating new files or when completely replacing small files.

2. **AI Documentation Folder**: Any LLM should consult the AI documentation folder (`./ai_docs/`) to obtain meta-information and further instructions. This folder contains:
   - `README.md`: Index and overview of available documentation
   - `tool_creation_guide.md`: Detailed guide on creating new tools
   - `integration_guide.md`: Guide for using file operation tools
   Refer to these files for tool creation guidelines, project architecture, and best practices.

3. **Thinking and Planning**: Use the Thought tool to reason through complex problems before taking action. Document your reasoning to help with debugging and understanding your thought process.

4. **Creating helper files**: If you need to write small files for testing or storing some temporary info, use the ./temp folder. Do not pollute the workspace

5. **File removal**: Instead of removing files, move them to temp so they are not lost before the user cleans manually. This is a safety measure

9. **Directory Structure Diagnosis**: When you start or encounter file access errors, immediately determine the project root location:
   1. **Initial Check**: Use FileLister on `.` (current directory) and `..` (parent directory).
   2. **Identify Project Root**: Look for key project files: `agent_core.py`, `tools/`, `ai_docs/`.

   3. **Adjust File Paths**:
      - If project root is `.`: Use normal file paths (e.g., `"agent_core.py"`).
      - If project root is `..`: Prefix all file paths with `../` (e.g., `"../agent_core.py"`).
    4. **Quick Verification**: Use FileEditor to read `../ai_docs/integration_guide.md` lines 1-10 to confirm file access.

10. **Git Worktree Architecture**:
    - The project uses a git worktree architecture with separate directories for stable and development branches.
    - There is no construction workspace; all file operations are performed directly in the current worktree.
    - Determine which worktree you are in by checking the project root (see Directory Structure Diagnosis above).
    - When developing, make changes in the development worktree; when ready, merge to stable worktree via git.
